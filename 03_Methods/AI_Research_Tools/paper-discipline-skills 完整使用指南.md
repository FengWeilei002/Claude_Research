# paper-discipline-skills 完整使用指南

## 概述

本笔记是 [ChenHaNing/paper-discipline-skills](https://github.com/ChenHaNing/paper-discipline-skills) 的完整使用手册，逐条说明 12 个 Skill 的触发时机、操作流程、关键模板和使用细节。

**核心前提**：所有 Skill 都是**自动触发**的——你不需要手动调用，AI 看到你的请求后会按 Skill 的 description 自行匹配。如果 AI 没触发而你觉得该触发，可以说"按 paper-xxx 走一遍"或直接输入 `/<skill-name>`。

## 1. paper-using-skills — 入口与强制触发器

### 触发时机
任何科研写作场景的第一次交互。这是整个 Skill 体系的"守门人"。

### 核心机制
这个 Skill 不执行具体操作，而是强制 AI 在接到你的任何论文相关请求时，先扫一遍「触发对照表」，判断是否有其他 Skill 需要调用。匹配标准是"1% 可能相关就调用"。

### 触发对照表（AI 视角）
AI 会按这张表对你的请求做初筛：

| 你的请求包含... | AI 调用 |
|---------------|---------|
| 新论文、新项目 | `paper-claude-md-bootstrap` |
| 改一下/整理/润色（模糊） | `paper-confirm-before-doing` |
| 顺便/另外/还有（多任务） | `paper-one-session-one-task` |
| 跨文件改/统一术语 | `paper-protect-terminology` |
| .docx/Word | `paper-backup-before-word` |
| 全部/批量/所有（≥ 30） | `paper-pilot-before-batch` |
| 批量引用/术语/格式核查 | `paper-parallel-audit` |
| 导师说/录音/批注 | `paper-translate-advisor-feedback` |
| 改核心论点/研究问题/结论 | `paper-logical-consistency` |
| 改完了/可以发了 | `paper-verify-before-handoff` |

### Skill 优先级
多个 Skill 同时命中时的执行顺序：
1. `paper-confirm-before-doing` — 先确认方案
2. 场景 Skill — 决定怎么做
3. `chinese-de-aigc` — 文段改写时走五步闭环（如有安装）
4. `paper-verify-before-handoff` — 交付前最后一道闸

### Red Flags（AI 内部自检信号）
以下念头出现时 AI 必须停止并去查 Skill：
- "这个任务很简单，直接做就行"
- "用户没说要走流程，我先做了再说"
- "时间紧，跳过流程更快"
- "我了解这个用户的习惯"
- "这次跟书里写的情况不太一样"

## 2. paper-claude-md-bootstrap — 新项目初始化

### 触发时机
- 你说"开始写新论文"、"新开一个项目"、"帮我写第 X 章"
- 当前工作目录没有 CLAUDE.md 或 CLAUDE.md 内容 < 30 行
- 你说"我有篇论文…"而 AI 不知道项目背景

### 操作流程（AI 执行顺序）
1. `ls` 检查目录是否有 CLAUDE.md
2. 没有 → 停下当前任务，告诉你先建 CLAUDE.md
3. 逐项问 6 个问题（一次问一个，不一次列六个）
4. 全部答完后 Write 落盘 CLAUDE.md
5. 回到你最初的任务

### 必问的六项
| # | 字段 | AI 会问 |
|---|------|--------|
| 1 | 研究主题 | "一句话说清这篇论文研究什么、回答什么问题" |
| 2 | 投稿目标 | "投哪个期刊/会议？字数限制？格式？截稿日？" |
| 3 | 术语保护清单 | "哪些专业术语/缩写必须保留？防止我同义替换" |
| 4 | 文件结构 | "主稿在哪？参考文献在哪？图表在哪？" |
| 5 | 写作风格 | "第一人称还是第三人称？主动还是被动？紧凑还是详尽？" |
| 6 | 导师偏好 | "导师特别在意什么？逻辑线/引文密度/图表规范？" |

### 对 ResearchVault 的意义
当前 vault 已有根目录 CLAUDE.md，但如果你为某篇具体论文创建子项目（如在 `02_Thesis/` 下），那个子目录可能缺少项目级 CLAUDE.md。此时这个 Skill 会自动介入。

## 3. paper-confirm-before-doing — 先确认方案，再动手

### 触发时机（命中任一即触发）
- **宽泛动词**：改、整理、润色、重写、优化、清理、统一、规范化、处理一下、看一下
- **范围未指定**：没说改哪几节、哪几段、哪几个文件
- **方法未指定**：没说按什么标准改、保留什么、删掉什么
- **量词模糊**："全部"、"所有"、"都"但不明确指哪个集合

### 强制流程（AI 必须执行）
```
收到模糊任务 → 立即停（不调用任何 Edit/Write/Bash 写操作）
→ 回复四件事：
    1. 我理解的任务边界（改哪里、不改哪里）
    2. 我打算用的方法（标准、保留与删除原则）
    3. 一个最小样本（先改一段给你看）
    4. 显式问句："方案行的话就推全文，可以吗？"
→ 等待你回复
→ 你确认 → 按方案执行
→ 你调整 → 改方案后重新确认（不是直接改完给你）
```

### 标准回复模板（AI 会套用这个格式）
> 我理解的任务是：**[具体范围]**。
> 准备这样做：**[方法、标准、保留与删除原则]**。
> 不会动的部分：**[显式列出]**。
> 先给你一个小样本：**[一段/一节/一处示意]**。
> 你看方案行的话，我就按这个推全文。可以吗？

### 可以跳过确认的例外（仅三种）
1. 你明确写出了完整方案（"把这五段里的『论证』改成『论述』，其他不动"）
2. 纯查询类（"这段在哪个文件里"、"这个引用对不对"）
3. 回答你的提问（不涉及写文件/跑批量）

### 实际使用感受
这是所有 Skill 中使用频率最高的。你会发现当你随口说"帮我整理一下这段"，AI 不会再直接动手，而是先跟你确认它的理解。虽然多了一步，但避免了"改完发现不对"的返工。

## 4. paper-one-session-one-task — 一次会话只做一件事

### 触发时机
- 你用了"顺便"、"另外"、"还有"、"再帮我"、"对了"
- 一条消息里包含两个以上独立的写作/修改/核查任务
- AI 正在做任务 A，你临时追加任务 B

### 强制流程
```
检测到多任务 → 立即停 → 列出所有任务，标号
→ 回复："这看起来是 N 件独立的事。
  为了不让它们互相污染上下文，建议这次会话只做其中一件。
  你想先做哪一件？其他的我们下次开新会话。"
→ 等你选 → 只做被选中的那件
```

### 可以同会话连做的例外
1. **强依赖**：A 的输出是 B 的输入
2. **同一对象的多步骤**：同一份文件的 grammar → 术语 → 风格（属于一件事的内部步骤）
3. **5 分钟内的小操作**：补一个引用、改一个错别字

### 使用建议
这个纪律对 Claude Code 尤其重要——上下文窗口越长，AI 的判断力越差。如果你有 3 件事要做，开 3 个独立会话的质量远高于在一个会话里串行做完。

## 5. paper-protect-terminology — 术语保护

### 触发时机
- 跨多文件批改、整篇/整章润色/改写
- 要求"统一"某类表述
- 涉及段落里出现：模型名、算法名、自定义术语、缩写、人名、专业概念
- 用了"规范化"、"标准化"、"通顺一下"等动词

### 强制流程
```
检测到触发 → 扫描相关段落，提取所有候选保护术语
（缩写/模型名/算法名/自定义概念/人名/关键专业词）
→ 读 CLAUDE.md，把"术语保护清单"里的词加进来
→ 列出清单："下面这些词我不会动。有遗漏请补充；有不该保护的请删除。"
→ 等你确认/调整
→ 按确认后的清单执行
→ 完成后核一遍：被保护词数量是否变化？
```

### 六条扫描规则
1. **缩写**：连续 2 个以上大写字母（如 NLP、CRF、BERT-base）
2. **模型名**：英文+数字组合（如 GPT-4、Llama-2-70B）
3. **复合术语**：连字符或下划线连接（如 cross-attention、in-context-learning）
4. **首字母大写名词**：可能是人名或专有名词
5. **重复出现 ≥ 3 次**的非通用名词
6. **CLAUDE.md 术语清单**中已声明的所有词（最高优先级）

### 对你的研究场景
运筹学/经济学论文中需要保护的术语示例：
- 特定模型名：Stackelberg game, Nash bargaining, VCG mechanism
- 专业概念：operational flexibility（运营柔性）、supply chain coordination、mechanism design
- 自定义缩写：RES（renewable energy source）、PPA（power purchase agreement）
- 中文关键术语：机制设计（不要改成"制度设计"）、运营柔性（不要改成"运营灵活性"）

## 6. paper-backup-before-word — Word 操作前必备份

### 触发时机
- 修改/润色/改写/重排任何 .docx 文件
- 你提到 "Word"、".docx"、"文档"、"批注"
- "导师在 Word 里批注了，帮我改一下"

### 强制四步流程
```
第 1 步：创建备份
  cp "原文件.docx" "原文件.bak.YYYYMMDD-HHMMSS.docx"

第 2 步：判断是否值得转 LaTeX
  - 主稿大改/长期项目 → 建议转 LaTeX
  - 导师批注小修/一次性改 → 直接改 .docx

第 3 步：执行修改

第 4 步：交付时提醒
  "改完了，请你**先打开 Word 验证一下能正常打开**，
   有问题立刻回滚到 [备份文件名]。"
```

### 备份命名规范
```bash
cp "第三章.docx" "第三章.bak.$(date +%Y%m%d-%H%M%S).docx"
```
格式：`<原名>.bak.<时间戳>.docx`

禁止：
- `第三章_备份.docx`（无时间戳，下次覆盖）
- `第三章 (副本).docx`（系统默认名会冲突）
- 不留备份就开始改

### 转 LaTeX vs 直接改的决策表
| 情况 | 推荐 |
|------|------|
| 主稿、长期写作、还要改 ≥ 5 次 | 转 LaTeX |
| 导师在 Word 里批注，只改这一次 | 直接改 .docx |
| 涉及大量公式/表格重排 | 转 LaTeX |
| 涉及 Track Changes 修订/批注必须保留 | 直接改 .docx |
| 投稿要求必须 Word 模板 | 直接改 .docx，保留 LaTeX 草稿 |

## 7. paper-pilot-before-batch — 批量任务先试点

### 触发时机
- 你说"全部"、"所有"、"每一个"、"批量"、"统一"
- 操作对象数量 ≥ 30
- 数量未知（"整个目录的"、"全篇的"、"剩下的"）
- 涉及修改性操作（不是只读/查询）

### 强制流程
```
检测到批量 → 随机抽 3-5 个样本（不要只取前 3 个）
→ 只对样本执行 → 展示每个样本"改动前 vs 改动后"和"判断依据"
→ 问："这 N 个样本的改法，逻辑对的话推全量（共 M 个）。有不对的请指出。"
→ 确认 → 全量（≥ 30 时配合 paper-parallel-audit 并行）
→ 不对 → 改逻辑 → 重新跑样本
```

### 抽样策略（不是瞎抽）
- 1 个**最常见**的情况
- 1 个**边界情况**（最长/最短/含特殊符号/跨行）
- 1 个**完全不该被处理**的情况（验证不会误伤）

### 可以跳过样本的例外
- 纯只读（批量统计、列出）
- 逐条用户确认（实质不是批量）
- 数量 < 30 且每条 < 1 秒

## 8. paper-parallel-audit — 大批量核查的并行 Agent 模式

### 触发时机（全部满足才触发）
- 任务规模 ≥ 30
- 各条目之间无依赖
- 操作同质（每条用同样标准）
- 已通过 paper-pilot-before-batch 跑过样本

### 标准四组件模式
```
组件 1 — 分片
  total = 156 条 / shard_size = 50
  shards = [0:50, 50:100, 100:156]

组件 2 — 每个 Agent 独立输出 JSON
  {"shard_id": "0-50", "total": 50, "results": [...], "completed_at": "..."}

组件 3 — 主进程汇总 + 落盘
  python merge_shards.py shard-*.json > audit_report.json

组件 4 — 断点续跑
  任一 Agent 挂了只重跑那个分片
```

### 效率对比
- 156 篇引用串行核查 ≈ 5 小时 + 中间挂了从头来
- 3 Agent 并行 + JSON 落盘 ≈ 1 小时 + 任意挂了只重跑那个

### 不要并行的情况
- 条目之间有依赖（B 用 A 的结论）
- 任务规模 < 30
- 操作不同质（每条要不同处理）

### 对 ResearchVault 的应用
最适合的场景：文献综述中 50+ 篇论文的引用格式核查、术语一致性扫查、参考文献元数据补齐。

## 9. paper-translate-advisor-feedback — 导师反馈先翻译再执行

### 触发时机
- 你说"导师说…"、"老师让…"、"我导师反馈…"
- 提供录音转写文本、Word/PDF 批注、微信截图、邮件意见、手写便条

### 核心原则
导师说的是"人对人"的语言：省略、隐含、上下文依赖。AI 会按字面理解而抓错重点。你的工作是把导师的意图翻译成 AI 能执行的清单。

### 强制流程
```
收到导师反馈 → 逐条解析每条意见的五个要素：
  1. 位置（哪一章/节/段/句）
  2. 改什么（删/增/重写/调整顺序）
  3. 改成什么（目标状态）
  4. 标准（按什么判断改对了）
  5. 不动哪里（显式列出）
→ 位置不明 → 列"可能位置"问你
→ 意图不明 → 列"可能意图"问你
→ 输出清单等你确认 → 按确认后清单执行
```

### 翻译模板
| 字段 | 内容 |
|------|------|
| 原话 | "[导师原话]" |
| 位置 | 第 X 章 §X.X 第 X 段 / 需确认 |
| 改什么 | 删除/新增/重写/调整/补数据/换图 |
| 改成什么 | [目标状态，具体到文字/结构] |
| 标准 | [字数/风格/数据要求/引用要求] |
| 不动 | [显式列出] |

### 典型场景
导师批注："这块逻辑不顺，第二段和第三段有点跳。"

AI 不会直接把这句扔进 prompt，而是翻译成：
- **位置**：§3.2 第二段、第三段
- **改什么**：补过渡句，把第二段"已有研究的局限"和第三段"我们的方法"接起来
- **改成什么**：第二段末尾加"因此，本研究试图解决 X"
- **标准**：过渡句 ≤ 30 字，不引入新概念
- **不动**：第二、三段核心论点和已有引文

## 10. paper-logical-consistency — 论证一致性

### 触发时机
- 改动研究问题、核心假设、主要结论、因果机制
- 改动方法描述（边界条件/关键参数/适用范围）
- 对已有研究/竞品的批评力度变化
- 改动数据的解释（不是数据本身）
- 交付前要求"检查一下逻辑"

### 与 paper-protect-terminology 的本质区别
| | protect-terminology | logical-consistency |
|---|---|---|
| 抓什么 | 术语字符没被改 | 概念语义保持一致 |
| 检查方式 | grep 字符串 | 阅读上下文判断语义 |
| 谁来判断 | AI（用 grep 数数） | 用户（AI 只列疑点） |
| 例 | "BERT"没被改成"伯特" | §3.2 说"XX 通过 ZZ 影响 YY"，§4.5 也是这个表述 |

### 强制流程
```
检测到核心声明改动 → 提取"核心声明锚点"：
  - 研究问题/假设的关键短语
  - 因果关系的主词谓词宾语
  - 方法的边界条件/关键参数
  - 对已有研究/竞品的态度词
→ grep + 语义同义词扩展 → 全文搜索锚点位置
→ 列出每个位置上下文片段（§X.X 第 Y 段 + 原文约 50 字）
→ 问你："这些位置提到同一概念，是否需要联动修改？"
→ 等你对每处给出 ✅同步改 / ❌保持原样 / ❓我看一下
→ 按决定执行
```

### 锚点提取规则
| 改动类型 | 抓什么作锚点 |
|---------|------------|
| 研究问题 | 关键短语（"是否影响"、"通过 X 影响 Y"） |
| 因果机制 | 主词 + 谓词 + 宾语三元组 |
| 方法边界 | 边界条件/适用范围/关键参数 |
| 主要结论 | 结论的核心断言（强/弱/条件） |
| 对已有研究态度 | 评价词（支持/反对/修正/扩展） |
| 假设 | 假设的强弱程度词 |

### 扫描范围
至少覆盖：摘要 + 引言 + 方法 + 结果 + 结论 + 讨论

## 11. paper-verify-before-handoff — 交付前最后一道闸

### 触发时机
- AI 准备说"改完了"、"完成了"、"可以提交了"
- 你问"是不是可以了"、"能发给导师吗"
- 一个章节/整篇论文的修改任务即将关闭

### 强制流程
AI 即将宣告"完成" → 立即停 → 按 10 项硬清单逐条核 → 全部 PASS 才宣告完成 → 任何 FAIL 列问题并说明"还差这几条"

### 10 项硬检查清单

| # | 项 | 查法 |
|---|-----|------|
| 1 | **术语一致性** | grep 本次涉及的术语，确认没出现替换 |
| 2 | **引用完整性** | 所有 \cite{}/[X] 在参考文献里都能查到 |
| 3 | **数据正确** | 改动中的数字/百分比/年份与原始一致 |
| 4 | **图表编号** | 图 1、图 2…连续，文中引用全部命中 |
| 5 | **交叉引用** | "如 §3.2 所述"、"见表 4"全部命中 |
| 6 | **字数/格式** | 满足 CLAUDE.md 声明的要求 |
| 7 | **Todo 残留** | grep TODO/FIXME/xxx/?? 全部清掉 |
| 8 | **AIGC pattern** | 涉及文段改写时跑 chinese-de-aigc 五维自评 |
| 9 | **论证一致性** | 涉及核心声明改动时跑 paper-logical-consistency 复核 |
| 10 | **改动概要** | 一句话复述改了什么、未改什么 |

### AI 交付消息模板
每次交付时，AI 会输出类似：
```
改完了。交付前自查结果：
✅ 术语一致：[术语列表]，无替换
✅ 引用完整：新增 X 条引用都在参考文献里
✅ 数据：[关键数字]，与原始一致
✅ 图表编号：连续，引用全部命中
✅ 交叉引用：全部命中
✅ 字数/格式：当前 X 字/限制 X 字
✅ Todo：无残留
✅ AIGC pattern：本轮未涉及文段改写，跳过
✅ 论证一致：本轮未改动核心声明，跳过

本次改动概要：[一句话]
未触及部分：[一句话]

现在可以发给导师了。
```

## 12. paper-writing-discipline — 新坑变新 Skill

### 触发时机
- 你说"以后…都要…"、"下次记得…"
- "这种情况要先…"
- "帮我把这个习惯固化下来"
- "加一条新规则"

### 4 个判断题（不全过不写 Skill）
```
Q1：不用会翻车？
  ├─ 不会 → 写进 CLAUDE.md
  └─ 会 → 进入 Q2

Q2：是反直觉的？（必须经历才懂）
  ├─ 本来就该这样 → 太弱，不写
  └─ 必须踩坑才懂 → 进入 Q3

Q3：能在动手前触发？
  ├─ 只能事后补救 → 做 hook，不是 skill
  └─ 能在需求提出时触发 → 进入 Q4

Q4：是 HOW 不是 WHAT？
  ├─ 是工具用法 → 写文档/教程
  └─ 是做事方式 → 做 skill ✅
```

### Skill 命名规范
- 前缀统一 `paper-`
- 动词开头描述触发场景
- 小写字母 + 数字 + 连字符

| ✅ 好 | ❌ 坏 |
|------|------|
| paper-backup-before-word | word-backup-system |
| paper-pilot-before-batch | batch-task-helper |
| paper-confirm-before-doing | task-confirmation |

### 每个 Skill 必备 6 栏目
1. 核心理念（1-2 句 + 金句）
2. 触发条件（具体列表）
3. 强制流程（流程图/步骤）
4. 标准回复模板（能直接套用）
5. Rationalization Table（≥ 4 条，封掉合理化借口）
6. Red Flags（念头/动作级别）

### description 写法规则
description 写"**什么时候触发**"，不写"**做什么**"。

| ✅ 好 | ❌ 坏 |
|------|------|
| Use when 用户编辑 .docx 文件之前 | 这是一个 Word 备份工具 |
| Use when 用户给出模糊润色任务 | 用来润色论文段落 |

原因：description 进入 Claude 的 system prompt，决定它"要不要打开这个 skill"。写功能描述 → Claude 不知道何时调用 → 永远不会触发。

## 在 ResearchVault 中的推荐使用方式

### 日常文献阅读和研究笔记
大部分操作不直接触发这些 Skill——它们主要针对**论文写作和修改**场景。

### 论文章节起草时
会自然触发 `paper-confirm-before-doing`（AI 先确认你的意图再写），如果项目目录 CLAUDE.md 信息不全，还会先触发 `paper-claude-md-bootstrap`。

### 批量文献核查时
50+ 篇引用核查会自动走 `paper-pilot-before-batch` → `paper-parallel-audit` 的流水线。AI 会先拿 3-5 条引用试点，确认核查标准正确后，再分片并行跑全量。

### 导师反馈处理时
把导师邮件/批注/录音发给 AI 时，会触发 `paper-translate-advisor-feedback`，先把导师的口语化为可执行清单。

### 准备提交时
任何"改完了"会触发 `paper-verify-before-handoff` 的 10 项自查——术语有没有被偷换、引用有没有悬挂、交叉引用有没有断裂。

### 踩新坑时
如果遇到某个运筹学/经济学特化的翻车场景（比如优化模型符号不一致、博弈论假设在不同章节矛盾），可以用 `paper-writing-discipline` 创建专属 Skill。
